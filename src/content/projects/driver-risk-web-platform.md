## 문제

운수종사자의 인지검사·사고 이력 데이터로 사고 위험을 사전 식별하는 시스템이 필요했습니다. 출발점은 2025년 같은 주제의 경진대회 솔루션(상위 7.8%)이었지만, 대회 노트북과 운영 시스템 사이에는 큰 간격이 있습니다. 데이터 적재부터 학습, 버전 관리, 예측, 해석, 화면, 그리고 인터넷이 없는 환경에 설치되는 배포까지를 하나의 납품물로 만들어야 했고, 이 전 구간을 PL로 수행했습니다.

## 아키텍처

Nginx 뒤에 세 프로세스를 두는 구조입니다. React(Vite·Mantine) 관리자 웹은 정적 파일로 서빙하고, 예측·학습·SHAP·데이터 관리는 FastAPI AI 엔진이 전담하며, Spring Boot(eGovFrame) 백엔드는 공공 표준프레임워크 요건을 맞추는 최소 구성으로 두었습니다. 실질적인 무게중심은 AI 엔진입니다. 운영 DB는 SQLite(WAL)이고, 학습 산출물을 파일이 아니라 DB BLOB으로 저장해 모델 버전 목록·활성 전환·롤백을 관리자 화면에서 처리하도록 했습니다. 학습·추론이 오래 걸리는 것을 감안해 프록시 타임아웃을 2시간으로 잡고, 대용량 Excel 업로드 제한을 풀었습니다.

## ML: 두 계열 앙상블과 누수 방지

라벨부터 설계했습니다. 사고 심각도 가중합을 검사 시점 이후 24개월 관찰 윈도우로 집계하고, 사망·중상 사고는 자동 양성, 나머지는 연령대×검사유형 코호트 내 90분위 초과를 양성으로 정의했습니다. 모델은 두 계열입니다. 도메인(신규·자격유지검사)별 스태킹 계열은 HistGradientBoosting·XGBoost·CatBoost 7개 모델을 캘리브레이션 후 가중 결합하고, 시퀀스 계열은 검사 응답 시퀀스 피처로 도메인 통합 LightGBM을 학습합니다. 누수 방지가 핵심이었습니다. StratifiedGroupKFold의 그룹을 운수종사자 식별자로 잡아 같은 사람이 train/valid에 걸치지 않음을 매 fold 단언으로 강제하고, 개인·코호트 사전확률 피처는 각 검사 시점 이전 기록만으로 계산했습니다. 앙상블 가중치와 온도는 0.5·(1−AUC) + 0.25·Brier + 0.25·ECE 복합 점수를 최소화하도록 탐색했고, 최종 점수는 두 계열의 고정 가중 블렌딩입니다. 저장된 학습 결과 기준 AUC는 스태킹 A 0.7277, B 0.7268, 시퀀스 0.7421입니다(실데이터는 비공개라 레포만으로는 재현되지 않습니다).

## 관리자 웹

대시보드(월별 추이·등급 분포), 운전자 목록(필터·검색·관리자 전용 언마스킹), 개인 진단(SHAP 위험 요인 분해와 PDF 리포트), 그룹 비교 분석, 대량 리포트 내보내기 화면으로 구성했습니다. SHAP은 TreeExplainer를 도메인·모델·폴드별로 캐시하고, 전역 분석은 서버 캐시에서 식별자로 조회하도록 바꿔 대량 요청의 전송량을 크게 줄였습니다. 재학습은 업로드 → 라벨 생성 → 두 계열 학습 → 리로드를 백그라운드로 오케스트레이션하고, 단계별 취소와 서버 재시작 시 고아 상태 정리까지 처리합니다.

## 폐쇄망 배포: 도커 없이 zip 하나로

납품 환경은 인터넷이 차단된 폐쇄망이라 처음부터 오프라인 설치를 전제로 설계했습니다. 포터블 Python 임베더블·JRE·Nginx와 의존성 wheel 전체를 zip 하나(약 400MB)에 담고, 설치 스크립트는 `pip install --no-index --find-links wheels`로 외부 네트워크 접근 없이 끝납니다. 빌드 스크립트는 검증된 이전 배포본 위에 새 산출물만 덮어쓰는 템플릿 오버레이 방식이고, DB·환경 파일·캐시가 zip에 섞이면 빌드를 중단시키는 잔여물 검사 게이트를 두었습니다. 운영 제약이 설계를 여러 번 바꿨습니다. 설치 스크립트의 기존 폴더 자동 삭제는 백신이 랜섬웨어로 오인해 프로세스를 강제 종료하는 문제로 수동 삭제 안내로 회귀했고, 부팅 직후 자원 경합으로 첫 기동이 실패해 자동 시작에 60초 지연을 넣었습니다.

## 실무 적용

행안부·NIA·한국교통안전공단 위탁사업 산출물로 납품해 실제 배포까지 진행했습니다. 배포 후 재학습을 전제로 한 완전 재설치 방식, 관리자 계정 주입, 방화벽 등록 같은 운영 절차를 설치 문서로 함께 전달했습니다.

## 배운 것

모델보다 배포 환경이 설계를 결정하는 경우가 많았습니다. 백신, 관리자 권한, 부팅 자원 경합처럼 코드 밖의 제약이 설치 스크립트의 형태를 정했고, "모델은 파일이 아니라 버전 관리 대상"이라는 원칙이 운영 화면(활성 전환·롤백)까지 이어졌습니다. 경진대회 솔루션을 시스템으로 만드는 일의 대부분은 모델 밖에 있었습니다.
