## 폐쇄망이라는 전제

한국 공공기관의 업무 환경은 망분리가 기본입니다. 인터넷이 차단되어 있고, 외부에서 만든 소프트웨어는 기관이 정한 매체와 절차로만 반입됩니다. 국가·공공기관에 도입되는 시스템은 유형에 따라 국가정보원의 보안적합성 검증 같은 사전 절차를 거치고, 정보시스템 구축에는 전자정부 표준프레임워크(eGovFrame)가 사실상 표준으로 요구됩니다.

운수종사자 사고 위험 예측 시스템을 위탁사업으로 수행하면서 이 전제를 처음부터 설계에 넣어야 했습니다. `pip install`도, `docker pull`도, Hugging Face 다운로드도 없는 환경에서 ML 모델과 웹 서비스 전체가 돌아가야 합니다.

## 실제 배포 목적지: 폐쇄망 클라우드

이 시스템의 실제 운영 배포는 발주기관이 운영하는 쿠버네티스 기반 폐쇄망 클라우드 환경에 했습니다. 인터넷이 닿지 않는 클라우드에서는 배포의 중심 문제가 "무엇을 어떻게 반입할 것인가"로 바뀝니다. 외부 레지스트리와 패키지 저장소가 전부 차단되므로 이미지·의존성·모델 같은 산출물을 기관의 절차에 맞는 형태로 준비해 들여보내야 하고, 들어간 뒤에는 내부 자원만으로 기동과 갱신이 이루어져야 합니다.

폐쇄망에 쿠버네티스 기반 플랫폼을 구축한 공개 사례들을 보면 공통 구조가 있습니다. 외부에서 이미지를 tar로 떠서 반입하고, 내부 사설 레지스트리에 적재한 뒤, 클러스터가 그 레지스트리만 바라보게 만드는 것입니다. 의존 차트·CRD·이미지까지 수십 개의 외부 파일을 사전에 전부 파악해 준비해야 한다는 점이 일반 설치와 다릅니다([공공 폐쇄망 K8s 플랫폼 구축기](https://www.makinarocks.ai/blog/%EA%B3%B5%EA%B3%B5-%ED%8F%90%EC%87%84%EB%A7%9D-%ED%99%98%EA%B2%BD%EC%97%90-k8s-%EA%B8%B0%EB%B0%98-ai-%ED%94%8C%EB%9E%AB%ED%8F%BC-%EA%B5%AC%ED%98%84%ED%95%98%EA%B8%B0/), [폐쇄망 배포 시스템 개발기](https://blog.banksalad.com/tech/how-we-have-built-alice/)). 반입이라는 관문 때문에 "밖에서 완성해 통째로 들여보내고, 안에서는 자급한다"는 원칙은 어떤 폐쇄망이든 같습니다.

발주기관 환경의 세부 구성은 기관의 자산이라 여기에 적지 않습니다. 이 글의 나머지는 공개할 수 있는 쪽, 즉 공개 저장소에 담은 설치형 트랙의 이야기입니다.

## 공개 저장소에 담은 것: 단일 PC 오프라인 설치형

공개 저장소는 이 시스템을 익명화해 재구성한 것인데, 배포 체계는 클라우드 의존 없이 **단일 Windows PC에서 인터넷 없이 전 구간이 재현되는 설치형**으로 담았습니다. 과제 중에 실제로 만들었던 트랙이기도 합니다.

폐쇄망 배포의 표준 답안처럼 통용되는 것은 도커입니다. 외부에서 `docker save`로 이미지를 tar로 뜨고, 반입해서 `docker load`로 되살리는 방식입니다([정리 글](https://velog.io/@hognod/Docker-Install-Offline)). 하지만 이 방식은 대상 머신에 도커가 이미 있고, 설치할 권한과 심의가 확보됐을 때의 답입니다. 단일 PC 트랙에서는 도커 자체가 설치 심의 대상이 되므로, 컨테이너 대신 **포터블 런타임 + 오프라인 wheel** 방식을 택했습니다. zip 하나(약 400MB)에 다음이 전부 들어갑니다.

- Python 임베더블 배포판, JRE, Nginx 바이너리 (설치 없이 압축 해제로 동작)
- 의존성 wheel 전체 + `get-pip.py`
- 빌드된 React 정적 파일, Spring Boot jar, FastAPI 소스
- 설치·기동·중지 스크립트와 운영 문서

설치 스크립트의 핵심은 한 줄입니다.

```bat
pip install --no-index --find-links "wheels" -r requirements.txt
```

`--no-index`가 PyPI 접근을 완전히 차단하므로, 이 설치가 성공한다는 것 자체가 "외부 네트워크가 필요 없다"는 증명이 됩니다. 인터넷에 나가지 않는다는 것을 말로 설명하는 것보다 이 플래그를 보여주는 편이 빠릅니다.

## 빌드: 템플릿 오버레이와 잔여물 게이트

400MB 번들을 매번 처음부터 만들지 않습니다. 빌드 스크립트는 **검증된 이전 배포 zip을 템플릿으로 받아**, 그 안의 런타임 번들(Python·JRE·Nginx·wheels)은 재사용하고 새로 빌드한 산출물(프론트 dist, jar, AI 엔진 소스)만 덮어씁니다. 런타임처럼 무겁고 잘 안 바뀌는 것과 애플리케이션처럼 가볍고 자주 바뀌는 것을 나눈 것입니다.

마지막 단계에는 잔여물 검사 게이트를 두었습니다. 출력 zip 안에 `__pycache__`, `*.db`, `.env`, 업로드 데이터, 모델 버전 디렉터리 같은 것이 하나라도 섞이면 빌드를 실패시킵니다. 개발 장비의 흔적이 배포물에 들어가는 사고는 폐쇄망에서는 회수도 어렵기 때문에, 사람의 주의가 아니라 스크립트가 막게 했습니다.

## 설치 스크립트가 마주친 현실

이 트랙을 개발하고 검증하는 과정에서 가장 많이 배운 부분은 화려한 기술이 아니라 설치 스크립트가 겪은 현실이었습니다.

**백신이 설치 스크립트를 죽인다.** 초기 버전은 재설치 시 기존 폴더를 자동 삭제했는데, 기업용 백신이 대량 파일 삭제를 랜섬웨어 행위로 오인해 스크립트 프로세스를 강제 종료했습니다. 몇 버전에 걸쳐 우회를 시도하다가 결국 자동 삭제를 포기하고 "운영자가 탐색기로 직접 삭제"하는 절차로 회귀했습니다. 기술적으로 후퇴처럼 보이지만, 동작하는 쪽이 정답이었습니다.

**부팅 직후는 자원 경합 지대다.** 로그인과 동시에 서비스를 띄우면 다른 시작 프로그램과 경합해 첫 기동이 실패하곤 했습니다. 작업 스케줄러 등록에 60초 지연을 넣는 한 줄로 해결됐지만, 이런 종류의 문제는 자동 시작 환경을 실제로 돌려 보기 전에는 재현조차 어렵습니다.

**관리자 권한으로 뜬 프로세스는 일반 권한으로 못 죽인다.** 중지 스크립트가 서비스를 종료하지 못하는 문제의 원인이었고, 스크립트가 스스로 권한 승격을 요청하도록 바꿔 해결했습니다.

기동 스크립트는 세 서비스(AI 엔진, 백엔드, Nginx)를 순차로 띄우며 각 HTTP 엔드포인트를 폴링해 준비를 확인한 뒤 다음으로 넘어갑니다. 원격으로 들여다볼 수 없는 환경에서는 "떴다"와 "준비됐다"를 구분하지 않으면 원인 파악이 한없이 길어집니다.

## ML 특유의 함정: 모델도 반입 대상이다

일반 웹 서비스와 달리 ML 서비스는 코드 밖 자산이 더 있습니다. 사전학습 모델과 임베딩입니다. Hugging Face 계열은 미리 받아 캐시를 반입하고 런타임에 `HF_HUB_OFFLINE=1`을 걸어 허브 호출을 차단하는 것이 표준이지만, 라이브러리에 따라 오프라인 플래그를 걸어도 원격 연결을 시도해 타임아웃으로 기동이 늦어지는 사례가 보고됩니다([fastembed 사례](https://github.com/qdrant/fastembed/issues/615)). 캐시 경로를 명시적으로 고정하고, 네트워크를 끊은 상태에서 기동 스모크 테스트를 해 보는 것이 확실합니다.

이 시스템은 부스팅 계열 모델이라 사전학습 모델 반입 문제는 없었지만 대신 다른 결정을 했습니다. 학습 산출물(모델·캘리브레이터·설정)을 파일 트리가 아니라 **DB BLOB으로 저장**해, 모델 버전 목록·활성 전환·롤백을 관리자 화면에서 처리하게 한 것입니다. 외부 모델 레지스트리를 쓸 수 없는 환경에서는 모델 버전 관리도 시스템 안에서 자급해야 합니다.

업데이트 전략도 환경에 맞췄습니다. 부분 패치 대신 **완전 재설치 + 재학습**입니다. 폐쇄망에서는 "지난 버전과 이번 버전이 섞인 상태"를 원격으로 진단할 수 없으므로, 상태 공간을 줄이는 쪽을 택했습니다.

## 폐쇄망 배포 체크리스트

클라우드형이든 설치형이든 공통으로 통하는 원칙을 체크리스트로 남기면 이렇습니다.

1. **반입물 전수 목록**: 이미지·런타임·wheel·모델·폰트까지, 다운로드가 필요한 모든 것을 빌드 시점에 번들
2. **오프라인 증명 가능성**: `--no-index` 같은 강제 장치로 "외부 접근 없음"을 코드로 증명할 것
3. **잔여물 게이트**: 배포물에 개발 흔적(DB·env·캐시)이 섞이면 빌드가 실패하게
4. **네트워크 차단 스모크 테스트**: 반입 전에 인터넷을 끊고 설치·기동·예측까지 통짜로 확인
5. **반입 후 자급**: 모델 버전 관리·롤백·로그까지, 외부 서비스 없이 시스템 안에서 해결되게
6. **환경의 마찰을 설계에 반영**: 백신, 권한, 부팅 경합은 코드 리뷰에 안 보이지만 실제 머신에서는 지배적
7. **운영 문서 동봉**: 설치·재설치·계정 교체·방화벽 절차를 배포물 안에

취약점 스캔이 요구되는 반입 심의라면 Trivy도 오프라인 모드(`--skip-update`, DB 사전 반입)로 준비해 두는 편이 좋습니다([공식 가이드](https://trivy.dev/docs/latest/advanced/air-gap/)).

설치형 트랙의 구현은 [driver-risk-prediction-platform](https://github.com/Minsu5452/driver-risk-prediction-platform)의 `deploy/`(build-package.sh, install.bat, nginx.conf)에 공개해 두었습니다. 백신 오탐이나 60초 지연 같은 이야기는 스크립트와 설치 문서에 그대로 남아 있습니다.
